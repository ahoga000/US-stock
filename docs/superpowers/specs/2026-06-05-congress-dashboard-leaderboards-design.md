# Congress Dashboard — Leaderboards 設計

**日期**：2026-06-05
**範圍**：只修改 `congress_dashboard.html`（單檔）。不動 `update.py`、不改 `data.json` 格式、不改爬蟲。

## 1. 背景與目標

現有儀表板以「議員」為中心：一張 203 位議員的表格，點開才看到個別持股。
使用者無法回答這些問題：

- 哪些股票被國會買進後績效最好／最差？
- 被買進／賣出最多的是哪幾檔？
- 哪些議員的報酬率最高、值得追蹤？

**目標**：在不丟棄現有議員明細表的前提下，新增三張排行榜，讓使用者「找出有用的股票 + 看出誰值得追蹤」。

## 2. 實作方式

**純前端聚合**：所有排行榜在瀏覽器端用 JS 即時從 `data.json` 的 `members[].recent` 計算。

理由：不改爬蟲、不改資料格式、現有資料立即可用；2000 筆交易的聚合運算對瀏覽器無感；改動集中在單一檔案，風險最低、可回退。

## 3. 資料來源與聚合範圍

聚合宇集 = 所有議員 `members[].recent[]` 的全部交易。
（`recent` 每人上限 25 筆，但本資料集 2000 筆分散在 203 人、平均約 10 筆/人，未被上限砍到，視為完整。）

每筆交易可用欄位：`ticker`、`type`、`amount`、`date`，以及補價後的 `return_pct`、`trade_price`、`current_price`、`hold_days`（部分交易無這些欄位 = 未補到價）。

**買賣別判定**（沿用現有邏輯）：`type` 字串（小寫）含 `buy`/`purchase` → BUY；含 `sell`/`sale` → SELL；其餘 → OTHER。

**ticker 有效性**：空字串、`N/A`、`--` 視為無效，排除。

**報酬率原則（全域）**：所有「報酬率」一律**只計 BUY** 交易，且該筆需有有效 `return_pct`（非 null/NaN）。SELL 與未補價交易不計入任何報酬統計。

## 4. 頁面結構

頂部維持現有 `stats-bar` 統計卡。其下新增 **Tab 列**，四個分頁，同時只顯示一個分頁內容：

```
📊 股票績效  |  🔥 熱門買賣  |  🏆 人物榜  |  👤 議員明細(舊)
```

- 預設開啟 Tab 1（股票績效）。
- Tab 狀態不持久化：重新整理回到 Tab 1。
- 現有的篩選列（議院/政黨/排序/搜尋）**只屬於 Tab 4**，切到其他分頁時隱藏。

### Tab 1 — 股票績效榜（只算 BUY）

對每個有效 ticker，蒐集其所有「BUY 且有 `return_pct`」的交易：

- **平均報酬%** = 這些交易 `return_pct` 的算術平均
- **買進筆數** = 這些交易的筆數（即計入平均的筆數）
- **議員數** = 這些交易中不重複的議員名稱數
- **最近日期** = 這些交易中最新的 `date`

**去雜訊門檻**：提供 `≥1 全部 / ≥2 / ≥3` 三檔「最少買進筆數」選擇器，**預設 ≥1（全部）**。每列都顯示買進筆數，供使用者自行判斷可信度，並標示「筆數越少越僅供參考」。

> 設計決策（2026-06-05）：原訂預設 ≥3，實測發現真正的負報酬股票多集中在 1–2 筆交易的 ticker（≥3 集合內僅剩 5 檔負報酬），導致「跌最多」幾乎等於「漲最多」反轉、撈不到輸家。故改為預設 ≥1 全部，讓使用者看得到真實輸家，再用三檔選擇器自行收緊到高信心集合。

**排序**：可切換「漲最多」（平均報酬 desc）/「跌最多」（asc）。預設漲最多。

**展開明細**：點任一股票列就地展開組成該平均的每筆 BUY（議員 ｜ 買入日期 ｜ 金額 ｜ 個別報酬，依日期新到舊）。沿用熱門榜的展開機制。

**欄位**：Ticker ｜ 平均報酬% ｜ 買進筆數 ｜ 議員數 ｜ 最近日期

### Tab 2 — 熱門買賣榜（依交易筆數）

左右並排兩張表：**被買進前 30**、**被賣出前 30**。

- 被買進：對每個有效 ticker 計 BUY 筆數，取前 30。
- 被賣出：同上以 SELL 筆數，取前 30。
- 每列欄位：排名 ｜ Ticker ｜ 筆數 ｜ 議員數（不重複） ｜ 買進平均報酬%（取自 Tab 1 同一 ticker 的 BUY 平均，純參考；無資料顯示 —）
- 點 ticker 列 → 就地展開「誰在買/賣這檔」：議員名 ｜ 日期 ｜ 金額 ｜ 報酬%（該方向交易明細）。

「買進平均報酬%」欄在兩張表都顯示同一份 BUY 平均（含被賣出表），欄名明確標示為「買進平均報酬」避免與賣出混淆。

註：此處的 ticker BUY 平均報酬，一律以該 ticker「全部有 `return_pct` 的 BUY 交易」計算，**不套用** Tab 1 的 ≥3 門檻（門檻只決定 Tab 1 是否「列出」該 ticker，不改變平均值本身）。因此被賣出榜中某檔即使 BUY 不足 3 筆，仍可顯示其參考平均報酬。

### Tab 3 — 人物榜（誰值得追蹤）

對每位議員，蒐集其「BUY 且有 `return_pct`」的交易：

- **平均報酬%** = 這些交易 `return_pct` 平均
- **計入筆數** = 這些交易筆數
- **勝率** = `return_pct > 0` 的交易數 / 計入筆數
- 政黨、議院沿用既有欄位

**門檻**：只列「計入筆數 ≥ 3」的議員（避免單筆僥倖洗到第一）。
**排序**：可排序，預設平均報酬 desc。
**欄位**：議員 ｜ 平均報酬%(BUY) ｜ 計入筆數 ｜ 勝率 ｜ 政黨 ｜ 議院
**互動**：點議員 → 切到 Tab 4 並展開該議員明細（以 `name` 比對定位）。

### Tab 4 — 議員明細

= 現有表格與展開明細，原封不動搬入此分頁（含篩選列、排序、搜尋、點開圖表）。

## 5. 共用細節與邊界情況

- 報酬率顯示：取一位小數 + `%`，正數綠、負數紅（沿用 `.ret-pos`/`.ret-neg`）。無資料顯示 `—`。
- `return_pct` 為 null/缺欄 → 不計入平均、不計入勝率分母。
- 門檻開關（Tab 1、Tab 3）：預設啟用（≥3）。
- 跨分頁定位 Tab 3 → Tab 4：以議員 `name` 字串比對（merge 階段 name 為唯一鍵）；切換後清空搜尋、捲動到並展開該列。
- 切換分頁時銷毀已開啟的 Chart.js 實例（沿用現有 `S.charts` 清理機制），避免 canvas 殘留。

## 6. 非目標（YAGNI）

- 不改爬蟲、不新增 `data.json` 欄位。
- 不計算 SELL 的「已實現報酬」。
- 不持久化分頁/篩選狀態。
- 熱門榜不提供金額排序版本（僅交易筆數）。

## 7. 驗證

靜態 HTML，驗證方式：

1. 在瀏覽器開啟，逐一切換四個分頁，確認皆正常渲染、無 console 錯誤。
2. **抽樣核對**：挑一個 ticker，手動在 `data.json` 數其 BUY 筆數與平均報酬，與 Tab 1/Tab 2 顯示比對一致。
3. 選一個 Tab 3 議員點擊，確認正確跳到 Tab 4 並展開對應議員。
4.（選用）寫一次性 Python 腳本獨立重算 Tab 1/Tab 3 前幾名，與畫面交叉驗證後刪除。
